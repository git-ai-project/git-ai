Issue #1657 — Windows console-window popups

Regression analysis across 1.5.8 → 1.6.0 → 1.6.1 · git-ai-project/git-ai · branch claude/issue-1657-investigation-ge2v0p

VERDICT  Partly a regression. The bare mechanism — a console-subsystem git-ai.exe launched per hook by a console-less agent — is pre-existing and unchanged since ≤1.5.8. But 1.6.0 reworked the Windows daemon launch (PR #1587) and that change is the most likely amplifier: it (a) neutralizes CREATE_NO_WINDOW by pairing it with DETACHED_PROCESS, and (b) sits in front of a daemon-startup path that fails silently on permission errors — so the daemon never comes up and every hook re-attempts the (visible) launch. That second defect is the concrete, fixable root cause.

The flag interaction that matters

Microsoft's CreateProcess documentation states that CREATE_NO_WINDOW “is ignored if … used with either CREATE_NEW_CONSOLE or DETACHED_PROCESS.”

So from 1.6.0 onward the daemon's window suppression no longer comes from CREATE_NO_WINDOW at all — it relies entirely on DETACHED_PROCESS. The same DETACHED_PROCESS flag was added to the daemon self-restart path too (src/daemon.rs). This is a real behavioral change in exactly the reworked path.

3 · How a latent issue becomes repeated popups

Two independent factors combine. Neither alone is new; their interaction in 1.6.x is.

4 · Root cause of the daemon-startup failure

The reporter's git-ai debug shows both “daemon startup blocked: lock held” and “Access is denied (os error 5)”. Those are the same event: the lock file under ~/.git-ai/internal/ was created by an elevated/other user, and the lock helper collapses every open error — including permission denied — into “held by another process.” This matches issue #1287 (“daemon startup blocked … when installed as Administrator/Root”).

Before — masking (utils.rs)

// any error → None → "lock held"
fn try_lock_exclusive(path) -> Option<File> {
  OpenOptions::new()
    .create(true).write(true)
    .share_mode(0)        // windows
    .open(path)
    .ok()  // EACCES looks identical
           // to a real sharing violation
}

After — classified (utils.rs)

enum LockAttempt { Acquired, Contended,
                   Inaccessible(io::Error) }

match open(path) {
  Ok(f)  => Acquired(f),
  Err(e) => match e.raw_os_error() {
    Some(32|33) => Contended,  // real lock
    _           => Inaccessible(e),
  }                          // EACCES = perms
}

The daemon startup path now reports the truth and tells the user how to recover, instead of pointing at a process that doesn't exist:

- daemon startup blocked: lock held at …\daemon.lock
+ daemon lock is inaccessible at …\daemon.lock (Access is denied. (os error 5)).
+ This usually means the file or its parent directory was created by a different or
+ elevated user (e.g. an Administrator/sudo install). Fix the ownership/permissions of
+ the ~/.git-ai/internal directory (or remove it) so your normal user account can write…

Scope & deliberately-deferred follow-ups

This fix targets the amplifier (factor B) — the highest-confidence, Linux-testable defect that directly explains the diagnostics. Two complementary changes remain, documented for a follow-up:

  • Console flash itself (factor A). A GUI-subsystem hook launcher (or making the checkpoint entry point not allocate a console) is needed to fully suppress the per-hook flash, since the top-level process is created by the agent and git-ai cannot set its own creation flags.
  • DETACHED_PROCESS + CREATE_NO_WINDOW. Re-evaluate the 1.6.0 flag combo so window suppression doesn't depend on a flag the OS ignores.

6 · Reproduction

The Docker harness reproduces the permission-masking failure on Linux (the cross-user analogue of the Administrator-install case): a root-owned lock file inside an unprivileged user's home, with no daemon running, so any “lock held” report is provably wrong.
